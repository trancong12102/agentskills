# Ownership & Borrowing: Advanced Patterns

## Interior Mutability

### Cell<T> — Copy types only, zero overhead

`Cell<T>` provides interior mutability for `Copy` types with no runtime cost —
no borrow tracking, no panics. Use `get()` and `set()`.

```rust
use std::cell::Cell;

let count = Cell::new(0usize);
count.set(count.get() + 1); // no borrow, no panic possible
```

Best for: counters, flags, generation numbers in single-threaded code.

### RefCell<T> — non-Copy types, runtime borrow checking

`RefCell<T>` tracks borrows at runtime. `borrow()` and `borrow_mut()` panic
if the borrow rules are violated.

```rust
use std::cell::RefCell;

let data = RefCell::new(Vec::new());
data.borrow_mut().push(42);

// WRONG: panics — borrow_mut while borrow is active
let r = data.borrow();
let w = data.borrow_mut(); // PANIC
```

**Defensive pattern:** use `try_borrow_mut()` when you can't guarantee no
outstanding borrows:

```rust
match data.try_borrow_mut() {
    Ok(mut guard) => guard.push(42),
    Err(_) => log::warn!("concurrent borrow detected"),
}
```

### OnceCell / OnceLock — write-once initialization

- `std::cell::OnceCell` — single-threaded
- `std::sync::OnceLock` — multi-threaded (`Send + Sync`)

```rust
use std::sync::OnceLock;

static CONFIG: OnceLock<Config> = OnceLock::new();

fn get_config() -> &'static Config {
    CONFIG.get_or_init(|| load_config_from_disk())
}
```

## Smart Pointers

### Rc<T> + RefCell<T> — single-threaded shared mutable state

The canonical pattern for trees, graphs, and observer patterns in single-threaded
code. `Rc` handles shared ownership; `RefCell` handles interior mutability.

```rust
use std::{rc::Rc, cell::RefCell};

type NodeRef = Rc<RefCell<Node>>;

struct Node {
    value: i32,
    children: Vec<NodeRef>,
}
```

**Watch for reference cycles** — `Rc` does not detect cycles; use `Weak<T>` for
back-references to prevent memory leaks:

```rust
use std::rc::Weak;

struct Node {
    parent: Option<Weak<RefCell<Node>>>,
    children: Vec<Rc<RefCell<Node>>>,
}
```

### Arc<Mutex<T>> — multi-threaded shared mutable state

```rust
use std::sync::{Arc, Mutex};

let shared = Arc::new(Mutex::new(Vec::new()));
let clone = Arc::clone(&shared);

std::thread::spawn(move || {
    clone.lock().unwrap().push(42);
});
```

**`Arc<RwLock<T>>`** when reads vastly outnumber writes — multiple concurrent
readers, exclusive writers.

### Box<T> — heap allocation

Use `Box` for:

- **Recursive types** — `enum List { Cons(i32, Box<List>), Nil }`
- **Large values** — moving 10KB structs on the stack is expensive
- **Trait objects** — `Box<dyn Trait>` for owned trait objects
- **Fixed-size slices** — `Box<[T]>` is thinner than `Vec<T>` (no capacity field)

```rust
// Convert Vec to Box<[T]> when size is final
let boxed: Box<[u8]> = vec![0u8; 1024].into_boxed_slice();
```

## Cow<'a, T> — Clone on Write

`Cow` defers cloning until mutation is needed. Accept `Cow` when a function
sometimes modifies borrowed data and sometimes doesn't.

```rust
use std::borrow::Cow;

fn normalize(s: &str) -> Cow<str> {
    if s.chars().all(|c| c.is_lowercase()) {
        Cow::Borrowed(s)                 // zero allocation
    } else {
        Cow::Owned(s.to_lowercase())     // allocates only when needed
    }
}

// In function signatures — accept impl Into<Cow<str>> for flexibility
fn process(input: Cow<str>) {
    let owned: String = input.into_owned(); // only clones if borrowed
}
```

## Pin<T> — Immovable Types

`Pin` guarantees that a value will not be moved in memory. Required for
self-referential types (e.g., async state machines generated by the compiler).

**You almost never need to create `Pin` manually.** It comes up when:

- Implementing `Future` by hand
- Working with self-referential structs
- Using FFI with pointers to Rust values that must not move

Use `pin-project` crate for safe field projection:

```rust
use pin_project::pin_project;

#[pin_project]
struct MyFuture {
    #[pin]
    inner: SomeInnerFuture,
    buffer: Vec<u8>, // not pinned — can be moved
}
```

**Rule:** if a struct contains `Pin`, use `pin-project` — never write manual
`unsafe Pin` projections.

## Lifetime Tricks

### Lifetime elision rules (memorize these)

1. Each reference parameter gets its own lifetime.
2. If exactly one input lifetime, it's assigned to all outputs.
3. If one input is `&self` or `&mut self`, its lifetime is assigned to outputs.

### Higher-Ranked Trait Bounds (HRTB)

Use `for<'a>` when a closure or trait must work for _any_ lifetime:

```rust
// Fn(&str) -> bool is sugar for:
fn call_with_ref<F>(f: F) where F: for<'a> Fn(&'a str) -> bool {
    let s = String::from("hello");
    assert!(f(&s));
}
```

### Variance

| Type          | Variance in 'a / T | Implication                              |
| ------------- | ------------------ | ---------------------------------------- |
| `&'a T`       | Covariant in 'a    | Longer lifetime substitutes shorter      |
| `&'a mut T`   | Invariant in T     | Must be exact — prevents unsound casting |
| `fn(T) -> ()` | Contravariant in T | Accepts supertypes                       |

**Gotcha:** structs with `&'a mut T` are invariant in T — the compiler won't
allow lifetime shortening even when it seems safe.
